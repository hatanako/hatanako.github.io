## 1.事件和委托的区别

### 委托

委托本质上是一个类型安全的函数指针，可以指向一个或者多个方法  
访问权限：公共或私有，可以直接被外部赋值或调用

### 事件

事件本质上是一个对委托的封装，提供了受控的“发布-订阅”模型  
访问权限：外部只能通过+=和-=来订阅/取消订阅，触发权限仅限于该声明事件的类的内部

|​区别点​|​委托（Delegate）​​|​事件（Event）​​|
|----------|---------------|-----------|
|​触发权限​|任何持有委托引用的代码均可调用Invoke()。|只有声明事件的类内部可以触发（调用 Invoke()）|
​空引用风险​|可能（需手动检查 null）|自动处理（event?.Invoke() 语法糖）|
|​封装性​|低（外部可修改委托链）。|高（外部只能通过 += 和 -= 操作）。|
​典型应用​|回调、动态方法调用、LINQ 查询|GUI 事件（如按钮点击）、观察者模式、解耦模块通信。|

​优先使用事件：  
当需要实现发布-订阅模型（如 UI 交互、模块解耦）时，事件能提供更安全的封装。  
​直接使用委托：  
当需要灵活传递方法（如回调参数）或手动管理多播调用时，委托更合适。

## 3.装箱拆箱

装箱是指把值类型转换为引用类型的过程，例如，把`int`类型装进`object`类型中。  
拆箱则相反，是指把引用类型转换成值类型，过程中会有一次强制转换。

## 5.Unity的GC机制，如何尽量避免

GC是一个垃圾回收机制，可以手动调用也可以自动调用，它负责回收一些没有再被用到的对象等等。  
每次触发GC时，可能会产生一定的卡顿，所以我们需要去尽量避免触发它。  

优化：

1.减少内存分配：避免在频繁调用的方法中创建临时对象（在方法体中`new`）,比如foreach循环产生的枚举器等等  
2.使用对象池：尽量减少频繁使用的对象的创建和销毁（子弹、特效、或者是大量相同的敌人）  
3.使用结构体替代类：结构体分配在栈上（所以我为啥会说结构体是引用类型TvT），需要注意装箱拆箱  
4.缓存静态变量：避免重复获取
5.使用StringBuilder（这是什么）：替代字符串拼接  
6.协程的WaitForSeconds：使用单例缓存，避免重复创建  
7.需要返回空集合时，尽量返回静态空列表，而不是使用new

## 6.如何优化DrawCall

Draw Call 是 CPU 向 GPU 发送渲染命令的过程。  
每次 Draw Call 都需要 CPU 准备数据、设置渲染状态，开销较大。  
优化目标是在保证视觉效果的前提下，尽量减少 Draw Call 次数。

### 静态批处理

原理：在构建时，将所有使用相同材质的静态物体合并成一个大的网格，并合并顶点数据。运行时只需一次 Draw Call 绘制整个合并后的网格。  
要求：物体必须标记为 Static，且材质完全相同。  
优点：大幅减少 Draw Call，无运行时开销。  
缺点：增加内存占用（合并后网格不能单独剔除）；只能用于不动的物体。

### 动态批处理

原理：Unity 在运行时自动将满足条件（顶点数少、材质相同）的动态物体合并，每帧重新计算合并网格并一次性绘制。

要求：

+ 顶点数小于 300（移动端 225）。
+ 材质相同（纹理、Shader 一致）。
+ 物体不能受镜像变换（如缩放为负）。

优点：自动处理运动物体。  
缺点：CPU 开销较大（每帧合并顶点），仅适用于少量小物体。

### GPU Instancing

原理：一次 Draw Call 绘制多个相同网格、相同材质的物体，通过实例化缓冲区传递每个物体的变换矩阵、颜色等属性。  
要求：材质启用 GPU Instancing，Shader 支持实例化。  
优点：支持大量重复物体（如草丛、森林），CPU 开销极低。  
缺点：每个物体的绘制属性有限（如不能有不同纹理）。

### 打大图集

原理：将多张小纹理合并成一张大图集，使原本需要不同材质的物体共享同一张纹理和 Shader，从而满足批处理条件。  
工具：Unity 的 Sprite Packer、TexturePacker 等。  
注意：需要确保 UV 不重叠。

### 减少渲染状态变化

排序：将相同材质的物体放在一起渲染，避免频繁切换材质。Unity 按材质排序，但可以手动控制渲染队列。

使用 SRP Batcher：URP/HDRP 中提供的优化技术，即使材质不同，只要使用相同的 Shader 变体，SRP Batcher 就能减少设置材质的开销。它通过持久化材质属性块，减少 CPU 向 GPU 传输数据的次数。

例：  
场景中有 1000 个相同的树：使用 GPU Instancing。  
场景中有大量静态建筑：使用静态批处理。  
场景中有大量运动的石块：如果石块顶点数少，使用动态批处理；否则尝试 GPU Instancing。  
UI 优化：将多个 UI 元素合并到同一张图集，并使用 Canvas 的合并渲染（Unity UI 会自动批处理相邻的相同材质 UI 元素）。

### SRP（可编程渲染管线）

SRP 是 Unity 2018 引入的一套框架，允许开发者通过 C# 脚本完全控制渲染流程。它位于底层图形 API（如 Vulkan、DirectX）之上，提供了渲染循环、相机剔除、绘制调用、后期处理等接口。

优点  
极致灵活：可以实现自定义的渲染特性（如特殊的光照模型、风格化渲染）。  
性能优化：仅包含项目需要的渲染步骤。

缺点  
开发难度大：需要深入了解渲染管线。  
维护成本高：需自行处理多平台兼容性。

### URP（通用渲染管线）

定义  
URP 是 Unity 基于 SRP 构建的官方现成管线，旨在提供高性能、跨平台的渲染解决方案，取代了旧版内置管线。

核心特性

+ 单Pass正向渲染：所有光照在一个 Pass 内计算，减少 Draw Call。
+ SRP Batcher：通过持久化材质属性块，大幅降低 CPU 开销。
+ Shader Graph：可视化编辑 Shader，快速创建材质。
+ 内置后处理栈：包括泛光、景深、色调映射等效果。
+ 2D 渲染支持：专为 2D 游戏优化的光照系统。
+ 跨平台：从移动端到高端 PC 都能良好运行。

工作原理
URP 将渲染流程划分为多个 Pass（深度、不透明、透明、后处理等），开发者可以通过“渲染器特性”（Renderer Features）插入自定义效果。它使用正向渲染（Forward Rendering），但进行了优化，如使用延迟着色的可选配置（URP 12+ 支持）。

与内置管线对比

性能：URP 通常更快，尤其 CPU 开销更低。  
光照：URP 支持每对象最多 8 个逐像素光，内置管线只有 4 个。  
材质：URP 使用 Lit 和 Unlit 等专有 Shader，与内置不兼容。

建议

新项目首选 URP，除非需要高级功能（如光线追踪，可选 HDRP）。  
对于低端移动设备，URP 提供了丰富的质量等级设置。

## KCP 协议实现方法详解

核心机制
KCP 的核心是在 UDP 数据包基础上封装了自己的数据报结构，主要包含以下机制：

+ 数据包格式：每个 KCP 数据包包含会话 ID、序列号（sn）、分片信息、时间戳、窗口大小等头部字段。  
+ 可靠传输：  
  + 发送端：将用户数据拆分成多个数据包（MSS 控制），为每个包分配唯一序列号，放入发送缓冲区并启动定时器。  
  + 接收端：收到包后发送 ACK（确认包），包含已接收的序列号。如果发送端超时未收到 ACK，则重传该包。  
  + 快速重传：如果发送端收到多个重复的 ACK（例如同一序列号被多次确认），则立即重传，不必等待超时。  
+ 流量控制：  
  + 滑动窗口：发送窗口大小受接收端通告的窗口（rwnd）和拥塞窗口（cwnd）限制。  
  + 拥塞控制：KCP 实现了类似 TCP 的慢启动、拥塞避免、快速恢复算法（可配置）。  
+ 有序交付：默认按序列号顺序提交数据，但也可配置为无序（实时性场景）。  
+ 分片重组：当用户数据大于 MSS（最大分段大小）时，KCP 自动分片，接收端根据序列号和分片信息重组。  

实现步骤（简化）  

1. 初始化：创建 KCP 对象，设置会话 ID，绑定底层 UDP 发送函数。  
2. 发送数据：调用 kcp_send() 将数据加入发送队列，KCP 自动分片并放入发送缓冲区。  
3. 驱动更新：周期性调用 kcp_update() 检查超时和发送窗口，触发实际 UDP 发送。  
4. 接收数据：底层 UDP 收到数据后，调用 kcp_input() 喂给 KCP，KCP 解析 ACK 和用户数据。然后调用 kcp_recv() 从接收缓冲区取出有序数据。  
5. 状态管理：维护连接状态（如连接、断开、等待），处理心跳。

优缺点

优点：比 TCP 延迟更低（尤其在丢包环境下），灵活可控，适合实时游戏。  
缺点：需要自己处理连接管理、加密、NAT 穿透等上层逻辑。

## 如何实现安全稳定的UDP通信

要实现安全稳定的 UDP 通信，需要从稳定性和安全性两个方面着手，通常基于 KCP 或 QUIC 等协议进行封装。

稳定性保障

+ 可靠性：采用类似 KCP 的 ARQ（自动重传请求）机制，确保数据包不丢失。
  + 确认重传：每个包都有唯一 ID，接收方回复 ACK，发送方超时重传。
  + 选择性重传：只重传丢失的包，而非全部。
+ 有序性：接收端按序列号重排乱序包，或根据应用需求选择无序交付。
+ 流量控制：防止发送方过快压垮接收方。通过[滑动窗口]（这是什么）通告接收缓冲区剩余空间。
+ 拥塞控制：根据网络状况动态调整发送速率（如 KCP 的拥塞控制算法）。
+ 连接保活：定期发送心跳包，检测连接是否断开，超时则关闭连接。
+ 丢包区分：区分拥塞丢包和随机丢包，避免盲目降低发送速率。

[滑动窗口]: https://zhuanlan.zhihu.com/p/650019037

UDP 本身不提供加密和防篡改，需在应用层实现：

加密传输：使用 DTLS（Datagram TLS）协议，它是 TLS 的 UDP 版本。可以集成库如 OpenSSL 的 DTLS 实现，或使用更轻量的 NaCl（Networking and Cryptography library）中的加密原语（如 crypto_secretbox 进行加密+认证）。  
完整性校验：每个数据包附带 HMAC（哈希消息认证码）或 AEAD 认证加密（如 AES-GCM、ChaCha20-Poly1305），防止数据被篡改。  
防重放攻击：在包中加入时间戳或递增计数器，接收端丢弃过期或重复的包。  
身份验证：连接建立时进行握手（类似 TLS 握手），交换证书或预共享密钥，确认双方身份。
